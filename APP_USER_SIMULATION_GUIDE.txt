📋 OVERVIEW
───────────────────────────────────────────────────────────────────────────────

The Application User Simulator creates realistic user personas across your 
entire lab infrastructure (Drupal, WordPress, Exchange, SharePoint, LDAP/AD).

Components:
  1. app_user_simulator.py (403 lines)
     - Creates realistic user personas (23+ users generated)
     - Assigns users to departments, roles, permissions
     - Generates application accounts across all platforms
     - Simulates user activity with realistic logging
  
  2. app_account_manager.py (328 lines)
     - Manages account lifecycles
     - Handles password policies per application
     - Tracks MFA, password expiration, account status
     - Generates audit reports
     - Pre-configured policies for each app

🔒 SECURITY CONSIDERATIONS
───────────────────────────────────────────────────────────────────────────────

1. Password Generation:
   - Uses Python secrets module for cryptographic randomness
   - Follows application-specific policies
   - SHA-256 hashing for storage
   - Never stores plaintext passwords

2. Account Lifecycle:
   - Tracks all account management actions
   - Enables audit trails
   - Supports password history validation
   - Prevents password reuse

3. MFA Support:
   - Per-application MFA configuration
   - MFA enablement tracking
   - Audit reporting on MFA status

4. Access Control:
   - Department-based segmentation
   - Role-based permissions
   - Group membership management
   - Site/resource-specific access

💡 TESTING SCENARIOS

Scenario 1: Credential Harvesting
  ├─ Use generated usernames from different departments
  ├─ Attempt different password combinations
  ├─ Monitor failed login attempts
  └─ Observe account lockouts

Scenario 2: Privilege Escalation
  ├─ Login as standard user

  ├─ Test permission boundaries
  └─ Try admin operations

Scenario 3: Lateral Movement
  ├─ Compromise one application (e.g., Drupal)
  ├─ Use credentials to access other services (Exchange, SharePoint)
  ├─ Test credential reuse across platforms
  └─ Map network from user's perspective

Scenario 4: Account Lockout Evasion
  ├─ Generate multiple users
  ├─ Distribute login attempts across accounts
  ├─ Test rate limiting effectiveness
  └─ Demo account lockout procedures
